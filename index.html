Follow the User Story Lifecycle diagram

When Lamar creates a User Story he puts it in the Product Backlog

at this point you may want to create a Change Ticket for this User Story, since you will need at least a 7 day window between the time the Change Ticket is created and when you install to Production

Steps to take for "Plan to an Iteration Box" on User Story Lifecycle diagram - Moving user stories from the Product Backlog to an Iteration as you begin working on them

2 different ways to do this 1) - Go to RTC - Go to your product backlog (Plans / All Plans - Product Backlog should be at bottom of list) - Hover over your User Story and drop down opens up to the left - On drop down choose Plan For - Either select an Iteration from the list - Or browse and choose Iteration from list and click on OK - Click on Save

2) - Go to RTC - Go to your product backlog (Plans / All Plans - should be at bottom of list) - Click on your User Story - On Planned For box click on the drop down - select an Iteration from the list - if you need to see more choices click on the 3 dots next to the open line - click on Save

Moving a User Story from one Iteration to another

- if your User Story is not finished by the end of the Iteration it is in, then you can move it and its Tasks that are not completed to another Iteration

- see the attached document Moving User Story and Tasks to the Next Iteration.docx

Set status to Dev Ready

Setting status of User Story to Dev Ready & Owning User Story

- Go to the Iteration you put your User Story in (Plans / All Plans)

- If you do not see your User Story on the Taskboard, click on the Refresh icon (2 arrows in a circle pattern on top right to the left of Save button)

- From the taskboard click on your User Story

- You will now be on the Overview tab of the User Story

- drop down the Owned By box and select your name

- drop down the Story Points box and choose your estimate of the size of this User Story (3, 5, 8, 13 or 20 - small to large)

- at the bottom of the Overview tab is the Discussion Box

- under this box is a box labeled "Add a comment..."

- click anywhere in this comment box and it opens up

- now enter - AppScan not applicable - this is not a Web-based application

- Click on Save (upper right corner of screen)

- the status for your User Story can be found on the upper right corner (below Save button)

- values for Status are on far left column of User Story Lifecycle diagram

- the status will probably be New or Define (if it is already Dev Ready then skip down to the next section - Steps to take for "Decompose into Tasks Box")

- drop down the status box and change the status until the status becomes Dev Ready (see below bullets)

- if the user story currently has a Define status, then drop down the status box and select Complete Definition Action

- click on Save (the status will change to Dev Ready)

- go to the next section Steps to take for "Decompose into Tasks Box"

- if the user story currently has a New status, then drop down the status box and select Start Defining Action

- click on Save (the status will change to Define)

- now drop down the status box and select Complete Definition Action

- click on Save (the status will change to Dev Ready)

- go to the next section Steps to take for "Decompose into Tasks Box"

Set Child Tasks to Done

Set Child Tasks under User Story to Done

make sure you have finished your testing in ALM (all Test Cases passed and UAT approval emails attached - see SLM ALM Steps document)

log back into RTC

Go to Plans / All Plans and select the current Iteration your User Story is in

Either click on the Task from the Task Board

Or click on the User Story from the Task Board

And then click on Links tab

And then click on the Task at the bottom of window under Children

Verifying Child Tasks under User Story are set to Done

2 different ways to do this:

1)

Go to the Iteration you put your User Story in (Plans / All Plans)

Click on User Story from Task Board

Click on Links tab

At bottom of window under Children, all of the tasks should now have a line through them indicating they are in a Done status

If not, click on the refresh icon (2 arrows in a circle pattern on top right to the left of Save button)

2)

Go to the Iteration you put your User Story in (Plans / All Plans)

Find the Child Tasks on the Task Board

They should all be under the Closed column now, indicating they are in a Done status

If not, click on the refresh icon (2 arrows in a circle pattern on top right to the left of Save button)

Attach User Story

Attach to the User Story, Stakeholder Approval of User Story

Send an email to the Stakeholder requesting approval to move the User Story to Production

On subject line reference the User Story number and name

Attach this email to the User Story by doing the following

Click on your User Story from the Iteration's Task Board

Click on the Links tab

Click on the box (Drop files to add them…) just below Attachments

Navigate to where you stored the Stakeholder approval email

Double click the email to attach it here

Click on Save

Next step is to request SQA to approve the User Story, but before doing that - see the checklist below

Checklist before requesting SQA to approve the User Story

Cover Status in ALM must state Passed (See SLM ALM Steps doc)

In ALM Test Lab you must have at least one Test Case and at least one UAT approval attached

In ALM Test Lab make sure all of the Peer Review questions have been answered

In RTC, make sure that all the child Tasks under the User Story are marked Done

In RTC, make sure you entered Appscan not applicable comment in Discussion box on Overview tab of User Story

In RTC, attached Stakeholder approval of User Story on Links tab

Requesting SQA Approval of User Story

Requesting SQA to approve the User Story

Click on the Approvals tab (on menu below Summary field)

If on this screen all you see is the text "Select an approval type to add approvals to your work item", then do the following (otherwise skip to next section)

Click Add Approval

Then type in SQA Approval in the name box

Click the Add Approvers link

In the Select Users Dialog search for SQA.

Click GM, I/S SQA - SQAapprover - Alternate_Billing, BCBST_OCM Team, Mobile

Click Add and Close

Click the Save Button.

Product Owner Approval

Product Owner Approval

You will get an email back from SQA when the User Story has been reviewed and approved

The SQA approver will change the status of the User Story to Verified

At this point you can IM Lamar to notify him that your User Story will need Product Owner approval

Make sure that you have gotten the Stakeholder approval to move this User Story to production (attached in Links tab of User Story)

Once Lamar does the Product Owner approval, he will change the status of the User Story to Production Ready

At this point you are ready to implement your User Story to Production

Follow the Defect Lifecycle diagram

Steps to take for "Defect is Created and Synced to RTC" on Defect Lifecycle Diagram

You may want to create a Change Ticket for this Defect now, since you will need at least a 7 day window between the time the Change Ticket is created and when you install to Production

Log into ALM for the product this Defect is in

Click on Defects folder (far left column)

Click on New Defect selection (first from left on top menu bar)

You will now be on the Details tab for New Defect

In Summary field give a short summary of the defect

In Description field give a more detailed description of the Defect

Drop down Record Type field and select Defect: Development

For the PPM Project field enter PS421737 for BlueAdvantage Enrollment or PS421756 for Membership Support Systems

Drop down Severity field and make the appropriate selection

Drop down Application field and select your application

Drop down Environment field and select Production

Drop down Workaround field and make the appropriate selection

Drop down Complexity field and select a choice

Now click on the Business Impact tab (just below the Summary description)

Enter the Business Impact

If Workaround was Yes, then click on the Workaround Details tab (just below the Summary description)

Enter the Workaround description

Click on Submit and the new defect will now be created

Click on Close, unless creating another Defect

Make note of the Defect ID number associated with this Defect, this will be referenced later in ALM when creating the test case in the Test Plan folder

The initial Status and Sub

Status of the Defect will be New / Pending Assignment

If you are ready to work on this Defect then you can change the Sub

Status to Repair In Process (you do not have to do this now, can wait until defect work is done and then change the Sub

Status)

If you want to change the Sub

Status now, then click on the Defect and on the Details tab, drop down the Sub

Status field and select Repair in Process then click on OK

It may also prompt you to add a comment (you can add a blank comment if you wish)

This will cause the Status field to change to Open

Test Plan (Create test cases and link them to a User Story or Defect)

Select Test Plan (far left column)

Choose the existing folder (or create a new folder) you want to put your test case under

You can reuse existing test cases from the Test Plan

Click on the existing test case, then skip down to either the

Req Coverage tab section if this is for a User Story

Linked Defects tab if this is for a Defect

To create a new folder, click once on the parent folder and click on new folder icon

New folder icon (first from left) is on top menu bar

Do the steps below for each Test Case you want to create for a specific User Story or Defect

Click on the beaker icon (second from left) on top menu bar to add a new test case

Details Tab

Enter the name of the Test Case in the Details window that pops up

Click on OK

Design Steps Tab

On the right side of the screen you are now presented with a new window with several tabs at the top (currently on the Details tab)

A Test Case is broken down into Design Steps (must have at least one design step)

Do the steps below for each Design Step you want to create for a specific Test Case

Click on the Design Steps tab

Click on shoe

asterisk icon (far left icon just below the tabs menu) to add a new step to this test case

Enter a description and expected result for this test step

Example for description is Run the TRR processing jobs

Example for expected result is The TRR processing job stream finishes successfully and the MERV records are applied correctly

Click on OK

Req Coverage Tab (if creating a Test Case for a User Story)

Click on Req Coverage tab

Click on Select Req (far left choice just below the tabs menu)

New window (Requirement Tree) to the far right appears

Click on the arrow just to the left of the Requirements folder (will open this folder)

Returns a list of User Stories

Click once on the User Story that the test case you are creating is for

Click on the Green Arrow (far left icon just below the Requirement Tree tab)

Now you have linked this Test Case to this User Story

Linked Defects Tab (if creating a Test Case for a Defect)

The Link Existing Defect window appears

Now you have linked this Test Case to your Defect

On the Defects folder in the left column for this Defect you will now see a chain link icon signifying you have linked a test case to this Defect

Refresh the screen if you don't see it (2 green circle arrows or PF5)

Execute Test Cases for a User Story or Defect

Select Test Lab (far left column)

Open the User Story or Defect folder to see the Test Sets underneath

Click on the Acceptance Testing Test Set that you want to use

In the right window, click on the Execution Grid tab

Click on the Test Case in the right window that you want to execute

Click on Run (top menu bar above test cases)

Manual Runner window pops up

Click on Begin Run (top menu bar)

Click on No Run under Status column and a drop down should now appear next to it

On the drop down select Passed

Now click on the square icon (End Run) at the far right of the top menu bar

If your Test Case has multiple Design Steps, each step must be run like above and set to Passed

If your Test Set has multiple Test Cases, each Test Case must be run like this

Attach UAT approval for User Story or EOT for Defect to Test Cases

User Story requires a UAT approval email attachment for each Test Case

Defect requires a EOT attachment for each Test Case, you do not need UAT approval

Click on the Test Set that you want to use

In the right window, click on the Execution Grid tab

Click on the Test Case in the right window that you want to add attachment to

Click on the paper clip icon (top menu bar above test cases)

Test Instance Details window appears

Click on the paper clip icon (on the Attach: menu bar)

Navigate to your UAT approval email or EOT and select it

The attachment should now show up in the Test Instance Details window

Click on OK

Now you are returned to the Test Case Window

There should now be a paper clip in the paper clip column for this Test Case

If your Test Set has multiple Test Cases, each Test Case must have a UAT approval attachment for a User Story or EOT attachment for a Defect

Execute Peer Review Test Set for a User Story or Defect

Select Test Lab (far left column)

Open the User Story or Defect folder to see the Test Sets underneath

Click on the Peer Review Test Set

In the right window, click on the Execution Grid tab

Peer Review Test Set window appears

Each row is a Peer Review question to be answered

For each row, click on the value in the Status column

On the drop down, select your answer (must be either Passed or N/A)

Be sure to click somewhere else off of the list of Peer Review questions when done

Otherwise the last question you were on may not reflect your update

Next steps after Testing is done (ALM Checklist)

In the Test Lab for your User Story or Defect

Check that your all your Test Cases have been Passed

For User Story, check that a UAT approval is attached to each Test Case

For Defect, check that a EOT is attached to each Test Case

Check that all of the Peer Review questions have been answered

Check that the Cover Status states Passed (this only applies to User Stories)

Select Requirements (far left column)

Select Requirements under this

Requirements window opens up to the right

Find your User Story in the list

Under the Direct Cover Status column, it should state Passed

If it does not, then you may not have Passed or executed all of your Test Cases

Creating the Release Work Item

Log into the RTC

No matter where you are in the RTC there is a menu bar at the top of the screen that has the choices Project Dashboards, Work Items, Plan, etc.

Click on Work Items

Click on 'Create from Template', found at the bottom of the left column in the display box returned

Select Release Work Item from the choices and then click on Next

Then click on Finish

Now click on the Release ID number that is returned

Drop down the Filed Against field and select your Product Project area

Drop down the Planned For field and select the Iteration you will be implementing this User Story or Defect in

Drop down the Deployment Lead field and select yourself

Drop down the Action field (below Save button and to the left) and select the Start Working Action

Click on Save

The Release status is changed to In Progress

Linking User Story(ies) or Defect(s) to the Release Work Item

Click on the Links tab (on menu below Summary field)

Click on Add Related drop down (bottom left of screen) and select Add Children

Select the User Story or Defect you intend to implement to Production from the list and then click on OK

You can link more than one User Story or Defect to this Release Work Item

Just select Add Children again and select the next User Story or Defect

If you do not see your User Story in the selection list, then that User Story is not in a Production Ready status

If you do not see your Defect in the selection list, then that Defect is either not in a Verified status or SQA has not approved it

Either have the Weekly Release moved farther out into the week, depending on others dependent on the Weekly Release

If can't be moved farther out, then you have the option of doing your own Change ticket / Release Work Item outside of the Weekly Release process

You can do this, even if your Change ticket window has passed, as long as your Change ticket has not been closed
